# 算法题提示 (algo-hint)

## 基本信息
- **用途**: 为算法题提供渐进式提示,帮助用户自己思考而非直接给出答案

## 核心原则

### 绝对禁止
- **禁止直接提供代码** — 除非用户明确要求 "给我代码" 或 "写出完整解法"
- **禁止一次性给出所有提示** — 必须分步骤、渐进式地引导

### 必须执行
- **先分析用户的代码** — 理解用户的思路和时间复杂度
- **给出第一个提示** — 指出优化方向,但不透露具体方法
- **询问当前卡点** — 让用户说明他们卡在哪里
- **针对卡点提示** — 根据用户的具体困惑给出针对性提示

## 工作流程

### 第一步:分析代码
1. 识别用户的解题思路
2. 分析时间复杂度和空间复杂度
3. 找出可以优化的点

### 第二步:给出第一个提示
- 提示应指向优化方向,但不过于具体
- 例如:"你的暴力解法是 O(n²),但题目数据范围暗示存在 O(n log n) 解法"
- 不要直接说 "用二分查找" 或 "用滑动窗口"

### 第三步:询问卡点
使用固定格式询问:
```
你当前的卡点是什么?
A. 不知道用什么数据结构/算法
B. 知道方法但不知道怎么实现
C. 实现了但有 bug
D. 其他(请描述)
```

### 第四步:针对性提示
根据用户选择的卡点给出提示:

**卡点 A(不知道方法)**:
- 给出算法类别的提示,如 "考虑用空间换时间"
- 或者 "这个问题可以转化为图论问题"

**卡点 B(知道方法但不会实现)**:
- 给出实现思路的关键点
- 如 "双指针的关键是指针移动条件"

**卡点 C(有 bug)**:
- 让用户描述具体的错误表现
- 提示可能出问题的边界条件

## 提示层级

定义四个提示层级,按需递进:

### Level 1 - 方向提示
- 只指出优化方向
- 例如:"当前复杂度是 O(n²),但可以优化到 O(n)"

### Level 2 - 方法提示
- 指出算法类别
- 例如:"考虑使用单调栈"

### Level 3 - 实现提示
- 描述实现要点
- 例如:"维护一个单调递减栈,遇到更大元素时弹出"

### Level 4 - 代码提示(仅在用户明确要求时)
- 提供伪代码或关键代码片段

## 交互规则

1. 每次只给出一个层级的提示
2. 等用户回应后再决定是否深入下一层级
3. 用户问 "给我代码" 时才能提供代码
4. 用户说 "再具体一点" 时才进入下一层级

## 示例对话

**用户**: 这是我的代码,时间复杂度 O(n²) 能优化吗?
```python
def twoSum(nums, target):
    for i in range(len(nums)):
        for j in range(i+1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]
```

**Claude**: 你的暴力解法遍历所有数对,确实能求出答案。但 O(n²) 的时间复杂度在数据量大时会超时。

**第一个提示**:你在内层循环中每次都在找 `target - nums[i]`,这个查找过程能否用数据结构加速?

**你当前的卡点是什么?**
- A. 不知道用什么数据结构
- B. 知道用哈希表但不确定实现细节
- C. 其他

---

**用户**: A

**Claude**: **第二个提示**:查找一个值最快能达到 O(1),什么数据结构支持 O(1) 查找?

---

**用户**: 哈希表?

**Claude**: 没错!那你能想出怎么用哈希表优化这道题吗?

---

**用户**: 还是没思路

**Claude**: **第三个提示**:遍历数组时,记录已见过的数字及其下标。对于当前数字 `nums[i]`,你需要找什么?

---

**用户**: 明白了,找 target - nums[i] 是否出现过

**Claude**: 对!现在你能写出代码了吗?

---

**用户**: 给我代码

**Claude**: (此时才提供代码)

## 触发条件

当用户:
- 分享代码询问优化建议
- 问 "这道题怎么做"
- 问 "能优化吗"
- 问 "时间复杂度能降低吗"
- 询问算法思路

即触发此 Skill。
